# Weekend Scout -- Design Changes Log

Changes from the original design (docs/weekend-scout-mvp-design.md), now superseded by docs/weekend-scout-design-v2.md.

| Date | Design Section | Change | Reason |
|------|---------------|--------|--------|
| 2026-03-26 | 5.2 SKILL.md | Added `disable-model-invocation: true` | Prevent auto-triggering during unrelated work |
| 2026-03-26 | 6. Project Structure | Skill lives in project .claude/ not ~/.claude/ | Keeps everything version-controlled together |
| 2026-03-26 | 4.1 Config | Using `platformdirs` for cross-platform config path | Windows compatibility (AppData vs ~/.weekend-scout) |
| 2026-03-26 | 5.2 SKILL.md | Removed `cd ~/weekend-scout &&` prefix from all CLI calls in SKILL.md | Package is installed via pip; `python -m weekend_scout` works from any directory |
| 2026-03-26 | 4.2 cities.py | `parse_geonames_file` returns `name_local` (native name) in addition to documented keys | Required by city list cache schema; also needed for local-script search queries |
| 2026-03-26 | 4.2 cities.py | `download_geonames` resolves `data/` via `Path(__file__)` rather than a fixed path | Works correctly in both editable install and when invoked from any working directory |
| 2026-03-26 | 4.2 cities.py | Query generation functions accept ISO date strings, not datetime objects as in design pseudocode | Consistent with CLI contract; avoids date object serialisation across process boundary |
| 2026-03-26 | 4.4 Query Generation | Query keywords are now language-keyed via QUERY_TEMPLATES (pl/de/fr/cs/sk/hu/uk/lt/lv/et/be/en) instead of Polish-only | Design pseudocode showed Polish example only; non-Polish configs need correct local keywords |
| 2026-03-26 | 4.4 Query Generation | `format_date_local` extended to use day-first order for fr/cs/sk/hu/uk/lt/lv/et/be | Manual test revealed these were incorrectly using English "Month Day, Year" order |
| 2026-03-26 | 4.4 Query Generation | English fallback query no longer appends year separately | `en_sat` from `format_date_local` already contains the year; removed duplicate |
| 2026-03-26 | 6. CLI | `main()` reconfigures stdout to UTF-8 on startup | Windows consoles default to cp1251/cp850 which can't encode Polish/Unicode city names in JSON output |
| 2026-03-27 | 5.2 SKILL.md | Full rewrite: explicit `suggested_queries.broad/targeted` usage, log-search pattern shown once for all phases, `format-message` CLI command replaces manual message template, `cache-mark-served` added to Step 6, `searches_this_week` dedup explained, event save schema documented | Multiple gaps between CLI data contract and skill instructions; previous version left targeted queries unused and events were re-sent every week |
| 2026-03-27 | 6. CLI | Added `format-message` subcommand to `__main__.py` exposing `format_scout_message` from telegram.py | `format_event_block` / `format_scout_message` were unreachable from CLI; skill had to replicate formatting manually |
| 2026-03-27 | 4.4 Query Generation | `generate_broad_queries` now returns `{"templates": [...], "vars": {...}}` instead of `list[str]`; `generate_targeted_queries` renamed to `generate_targeted_template(lang) -> str` (drops `tier1_cities`/`saturday` args) | Templates expose raw query patterns with `{placeholders}` so the skill can fill them for any city (including cities discovered mid-search, not just pre-known tier1 cities); separates locale-formatted date derivation (Python) from template filling (skill) |
| 2026-03-27 | 4.4 Query Generation / CLI init | `--city` override now geocodes coordinates + derives country/language from GeoNames file; added `COUNTRY_CODE_MAP` to config.py (ISO 3166-1 alpha-2 → country name); added `find_city_coords()` to cities.py | Previously `--city` only changed the city name string; coordinates stayed at the stored home city (Warsaw), so city list and queries were always Warsaw-based regardless of the override |
| 2026-03-26 | 4.2 cities.py | `parse_geonames_file` skips rows where `feature_code == "PPLX"` and returns `feature_code`, `admin2`, `admin3` per entry; `get_city_list` skips nearby PPL entries sharing admin codes with the home city | Seven European capitals (Warsaw, Brussels, Madrid, Paris, Dublin, Amsterdam, Stockholm) have city districts incorrectly tagged as plain PPL in GeoNames instead of PPLX, causing them to appear as Tier 1 cities. Two approaches were rejected: (1) `adm4 != ""` filter — unsafe because Germany populates adm4 for all 500 of its PPL entries (standalone cities like Halle, Hamm, Bremerhaven), and France/Belgium similarly; (2) minimum distance threshold — Warsaw's outer districts extend to 14.8 km while real standalone towns start at 9.5 km, creating unavoidable overlap. The implemented solution uses the home city's own `admin2`+`admin3` codes as the discriminator: any PPL entry within 15 km sharing those codes is a district of the home city (districts always share admin codes with their parent). |
| 2026-03-27 | 4.3 Telegram | Switched `parse_mode` from `"Markdown"` to `"HTML"`; replaced `_escape_markdown` with `html.escape()`; formatting via `<b>`, `<i>`, `<a href="">` | MarkdownV2 requires escaping 19+ characters (dash, dot, parenthesis, etc.) causing frequent `Bad Request: can't parse entities` errors with real Polish event names. HTML mode only requires escaping `<`, `>`, `&`, making it robust with arbitrary event data. |
| 2026-03-27 | New | Added `log_action()` to `cache.py` and `log-action` subcommand to CLI; all phases/lifecycle events now appended to `action_log.jsonl` in cache dir with `run_id`, `source`, `phase`, `target_weekend`, `detail` fields | Original design had no structured audit trail. After the first live run it was impossible to analyze search efficiency, phase coverage, or budget usage without adding action logging. |
| 2026-03-27 | 4.4 Query Generation | `QUERY_TEMPLATES`, `MONTHS`, and `format_date_local` expanded from 12 languages (Central/Eastern Europe) to 27 (added IT/ES/PT/NL/SE/NO/DK/FI/RO/HR/BG/RS/GR/TR/RU); `format_date_local` now uses module-level `_PERIOD_DAY_FIRST` set (de/no/da/hr/sr) instead of `lang == "de"` single-language check | Original design was Warsaw-centric. Auto-language detection via GeoNames (`cmd_init --city`) required correct templates for all supported countries. Norwegian/Danish/Croatian/Serbian share German's period date format ("10. april 2026"). |
| 2026-03-27 | 4.1 Config | `DEFAULT_CONFIG` no longer contains hardcoded country ("Poland") or coordinates (Warsaw 52.22/21.01); replaced with empty sentinels (`home_country: ""`, `home_coordinates: {lat:0.0, lon:0.0}`) | Clean-run test with Berlin: user ran setup but didn't know Berlin's coordinates, entered 0,0 → `get_city_list` computed haversine from null island → all European cities filtered out → city cache written with 0 entries. Poland/Warsaw defaults created a false "configured" appearance while silently producing wrong results. |
| 2026-03-27 | 4.1 Config | `run_setup_wizard` redesigned: now asks only city name and radius; auto-geocodes coordinates/country/language from GeoNames via `find_city_candidates()`; handles multi-country disambiguation with numbered list; falls back to manual coordinate entry only if city not found; removed country, lat/lon, and language prompts entirely; accepts `_geonames_path` parameter for test isolation | Original wizard had 8 prompts including lat/lon (users don't know their coordinates) and "Home city [Warsaw]:" default (confusing for non-Polish users). Language always derives from country; asking separately is redundant. |
| 2026-03-27 | 4.2 cities.py | New `find_city_candidates(city_name, geonames_path, country_filter=None) -> list` function: case-insensitive match on ASCII + native name, filtered to supported countries (COUNTRY_CODE_MAP), deduplicated by country (highest population), sorted by population descending | `find_city_coords()` already existed but returned a single best match globally, discarding country disambiguation. The new function returns one entry per country so the wizard (and skill) can handle ambiguous city names like "Lyon" (France vs. no other supported country match). |
| 2026-03-27 | 6. CLI | New `find-city` subcommand: `python -m weekend_scout find-city --name X [--country Y]`; returns `{"matches": [...]}` with name/country/lat/lon/language per entry; returns `{"matches": [], "warning": "..."}` when GeoNames file missing | Required for skill-driven setup: skill calls this to resolve the user's city to coordinates without spawning an interactive terminal wizard (which can't receive stdin in the skill's Bash tool context). |
| 2026-03-27 | 6. CLI | New `setup --json JSON` flag: merges provided key/value pairs into existing config, saves, and deletes stale `cities_<city>_<radius>.json` cache files when city name or coordinates change | Allows the skill to persist geocoded config (from `find-city` + WebSearch) without an interactive wizard. Cache invalidation prevents stale empty city lists (written with wrong coordinates) from being reused on the next run. |
| 2026-03-27 | 6. CLI / init | `cmd_init` now skips `get_city_list` entirely when `home_coordinates == {0.0, 0.0}` and appends `"warnings": ["coordinates_not_set: ..."]` to the output JSON; adds `home_country` to the config block | Prevents re-writing a stale empty city cache over a valid one; surfaces the misconfiguration to the skill so it can auto-fix without user intervention rather than silently running with no nearby cities. |
| 2026-03-27 | 5.2 SKILL.md | Step 1 setup flow: replaced "stop and tell user to run `python -m weekend_scout setup` in a terminal" with full in-chat onboarding. When `needs_setup: true`: asks user for city + radius, calls `find-city`, WebSearch fallback for unknown cities, multi-country disambiguation, then `setup --json`. When `coordinates_not_set` warning: runs the same resolution flow silently (city already known). | Entire onboarding now happens inside Claude Code's chat. User never needs a separate terminal. Also handles the partial-config case (city set, coordinates zero) that the previous guard missed. |
| 2026-03-27 | 5.2 SKILL.md | Step 5 Telegram handling updated: formatted message now always displayed to user in chat (HTML rendered as readable text); when Telegram not configured, shows exact `config` CLI commands with placeholder values; when configured but send failed, reports error with troubleshooting hint | Previous version silently skipped display when Telegram was configured, and gave no actionable guidance when unconfigured. First-time users had no way to know how to enable Telegram delivery. |
